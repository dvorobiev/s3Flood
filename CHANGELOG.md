## Changelog
### 0.12.0-beta — Двухпанельный браузер бакета

- **`s3flood browse --config config.yaml`** и пункт меню «⇄ Браузер бакета»: классический двухпанельный файловый менеджер (локальная ФС ↔ бакет) на prompt_toolkit — тот же стек и стилистика, что и остальной TUI.
- **Версии как навигация**: Enter на объекте открывает список его версий (дата, размер, короткий version-id, бейдж `⊙ latest`, delete markers); `..` — назад.
- **Операции**: `F5` — копирование между панелями (upload/download, в режиме версий — скачивание конкретной версии в файл `имя.<verid>`), `F6` — откат объекта к версии (server-side copy-object), `F8` — удаление объекта или версии, всё с инлайн-подтверждением y/n.
- Асинхронные вызовы `aws` CLI — интерфейс не блокируется; ошибки S3 показываются в статус-строке человекочитаемо.
- Новые модули: `s3browser_io.py` (слой данных, покрыт тестами на фикстурах) и `browser.py`; всего в проекте 102 теста.

### 0.11.0-beta — Достоверные измерения, отчёты v2 и новый интерфейс

**Методология измерений** (главное в релизе — цифрам теперь можно доверять):
- Исправлен расчёт перцентилей (раньше p90 мог оказаться ниже медианы); добавлены p50/p90/p95/p99 латентности отдельно для записи и чтения.
- Латентность больше не квантуется опросом процесса по 100 мс; при старте прогона замеряется оверхед холодного старта `aws` CLI (сотни мс на операцию) и указывается в отчёте (`client_overhead_ms`).
- CSV-метрики пишутся фоновым потоком (раньше файл открывался/закрывался на каждую операцию под глобальным локом, что сериализовало воркеров и искажало нагрузку).
- Датасет по умолчанию создаётся из несжимаемых данных со случайными размерами внутри групп (`--fill random`) — хранилища с компрессией/дедупликацией больше не показывают завышенные результаты. Старое поведение: `--fill zero`.
- Новая опция `warmup_sec`: операции первых N секунд выполняются, но исключаются из статистики.
- `duration_sec` в отчёте — активное время по меткам операций; wall clock вынесен в отдельное поле `wall_clock_sec`.
- Окно расчёта RPS больше не теряет операции при высокой нагрузке (было ограничение 400 операций).

**Отчёты**:
- `report.json` v2: блок `meta` (версия, время старта, hostname, конфиг прогона), `latency` с перцентилями, `errors` с разбивкой по типам (ServiceUnavailable, SlowDown, timeout, …), `timeline` — посекундные бакеты для графиков.
- Новые колонки CSV: `endpoint`, `thread_id`, `attempt`, `size_group`.
- Итог прогона печатается rich-таблицами (пропускная способность, латентность, ошибки) вместо сырого JSON.
- «Просмотр метрик» в меню — полноценный аналитический отчёт: сквозная скорость, спарклайн операций/с по времени, разбивка по группам размеров и endpoint, ошибки по типам, повторы, топ-5 медленных операций с меткой времени от старта. Понимает и старый формат CSV.

**Интерфейс**:
- Живой дашборд на `rich.Live` в btop-стиле: бейдж фазы (WRITE/READ/MIXED/WARMUP), endpoint и bucket в шапке, спарклайны W-RPS/R-RPS, прогресс с процентами, анимация активных операций. В CI/пайпе — компактная строка раз в 5 секунд.
- Рестайл меню: единый стиль questionary (указатель `❯`, акцентная подсветка), моноширинные глифы вместо эмодзи, бренд-панель с версией.
- Исправлен битый дефолтный endpoint `http://localhost:9000:9080` (двойной порт) в редакторе конфигов.

**Код**:
- `executor.py` разбит: `runner.py` (aws CLI, процессы, ретраи), `metrics.py` (статистика, CSV), `dashboard.py` (рендеринг).
- Появился набор тестов `tests/` (pytest, 75 тестов); устранены дубли и мёртвый код; чистка корня репозитория; README синхронизирован с фактическим поведением.

  - дефолтный endpoint для новых конфигов приведён к аккуратному виду: `http://127.0.0.1:9000`.
- **Унификация стиля интерактивного меню**:
  - курсор выбора приведён к единому виду (`»`), добавлены разделители и пиктограммы действий (💾, ⬅️);

### 0.9.10-beta — Исправления multipart upload и улучшения UI
- **Исправлена логика multipart upload**: теперь используется `multipart_threshold` из конфига вместо жестко заданного 5GB. Файлы >= `multipart_threshold` автоматически используют multipart upload через `aws s3 cp`.
- **Упрощена логика upload**: всегда используется `aws s3 cp`, который автоматически определяет необходимость multipart на основе настроек из переменных окружения.
- **Улучшения UI интерактивного меню**:
  - Убраны точки (индикаторы) из всех меню для более чистого вида
  - Добавлена проверка поддержки эмодзи с автоматическим fallback на текстовые метки
  - Убран лишний prompt "Нажмите любую клавишу для запуска..." если параметры не менялись
  - Исправлен спиннер при удалении объектов - убрано дублирующее сообщение
- **Вывод AWS CLI параметров**: в сводке параметров перед запуском теста теперь отображаются настройки multipart (threshold, chunksize, max_concurrent_requests) в читаемом формате (MB/GB).
- **Отдельный AWS config**: s3flood всегда создаёт временный профиль `s3flood-temp` в `~/.aws/s3flood-config` и не изменяет пользовательский `~/.aws/config`. Все параметры multipart/max_concurrent_requests берутся из конфига s3flood и действуют на всём протяжении теста.
- **Редактор конфигов**: убран дублирующий блок, значения отображаются списком, изменённые строки помечаются `*`, переключаемые поля (Yes/No, порядок, паттерн) меняются по одному Enter, а для endpoint'ов автоматически подставляется порт `:9080`. Цветовую подсветку временно упростили (вернёмся к ней в версии с полноценным TUI), inline-редактор и расширенный стиль добавлены в roadmap.

### 0.9.9-beta — Настройки AWS CLI параметров
- **Настраиваемые параметры AWS CLI**: добавлена возможность настройки `aws_cli_multipart_threshold`, `aws_cli_multipart_chunksize` и `aws_cli_max_concurrent_requests` через конфиг.
- **Удобный формат размеров**: размеры `aws_cli_multipart_threshold` и `aws_cli_multipart_chunksize` теперь можно задавать в MB (число, например `5120` для 5GB) или строками с единицами (например `"5GB"`, `"8MB"`). Числа без единиц интерпретируются как MB.
- **Приоритет над AWS конфигом**: настройки из конфига s3flood переопределяют настройки из `~/.aws/config` через переменные окружения `AWS_CLI_FILE_TRANSFER_CONFIG`.
- **Интерактивный редактор**: добавлена возможность редактирования AWS CLI параметров в меню «Редактировать существующий конфиг» с поддержкой формата MB/GB.
- **Динамическая конфигурация**: JSON-конфигурация AWS CLI формируется динамически из параметров конфига, с дефолтными значениями если параметры не заданы.

### 0.9.2-beta — Живой список Recent ops

### 0.9.0-beta — Переработанные профили нагрузки и порядок обработки файлов
- **Переработанные профили**: `write-heavy` → `write` (только запись), `read-heavy` → `read` (только чтение из бакета)
- **Профиль `write`**: только запись файлов из датасета в бакет, без фазы чтения
- **Профиль `read`**: только чтение объектов из бакета (в `/dev/null`), получает список объектов через `s3api list-objects-v2`
- **Порядок обработки**: добавлена опция `--order` (sequential/random) для выбора порядка обработки файлов
  - `sequential`: сначала маленькие файлы, потом средние, потом большие (по умолчанию)
  - `random`: случайный порядок обработки файлов
- **Улучшения**: обновлена логика отображения дашборда для новых профилей, улучшена обработка ошибок

### 0.3.2 — Fix BadDigest on Windows uploads
- Uploads `<5 ГБ` теперь выполняются одиночным `put-object` (мы поднимаем `multipart_threshold` через `AWS_CLI_FILE_TRANSFER_CONFIG`). Это устраняет `BadDigest` на S3-совместимых хранилищах без поддержки CRC32.

### 0.2.1 — фаза чтения в write-heavy профиле
- Профиль `write-heavy` теперь автоматически запускает фазу чтения после завершения всех upload-операций.

- Используется то же количество потоков, что и для записи.
- Прогресс-бар показывает метрики обеих фаз: запись (W) и чтение (R).
- Отдельные процентили latency для upload и download операций.
